# Handoff: Health Tracer — экраны Longevity · Workouts · Nutrition (+ все состояния)

## Overview
Три read-only «витрины» AI-коуча здоровья: **Динамика (Longevity)**, **Тренировки
(Workouts)**, **Питание (Nutrition)**. Это тёмная тема data-мира продукта. На этих
экранах пользователь **ничего не редактирует** — он смотрит свой недельный обзор и
активные планы, а любые изменения инициирует через **Чат** (коуч предлагает → человек
принимает). Выполнение (отметить тренировку, залогировать еду/воду/самочувствие) живёт
на отдельном экране **Сегодня**.

Ключевая фича этого пакета: **тренировки и рецепты можно смотреть как видео** — карточки
с постером, кнопкой play и длительностью, плюс полноэкранный плеер. Это отвечает на
вопрос пользователя «что именно делать».

## About the Design Files
Файлы в `design/` — **дизайн-референсы в HTML + React (in-browser Babel)**. Это прототипы,
показывающие задуманный вид и поведение, **не продакшн-код для копирования один в один**.
Задача — **воссоздать эти экраны в вашем кодовом окружении** (React/Next, Vue, SwiftUI,
native и т.п.) на его готовых паттернах, компонентах и системе стилей. Если окружения ещё
нет — выберите стек (рекомендация: **React + TypeScript + CSS-переменные / Tailwind**) и
реализуйте дизайн в нём.

Начните с переноса **токенов** (`app/kit.jsx`: объекты `L`, `D`, `M`) и **атомов**
(`Icon`, `Card`, `Chip`, `Btn`, `Ring`, `Progress`, `Eyebrow`, `CardHead`, `Avatar`,
`CheckCircle`), затем — **примитивов состояний** (`app/states.jsx`), и только потом
собирайте сами экраны (`longevity.jsx`, `workouts.jsx`, `nutrition.jsx`).

## Сквозные продуктовые правила (НЕ нарушать — они задают характер UI)
1. **Read-only витрины.** Минимум кнопок-действий «на месте». Акцент на данных и
   читаемости, а не на «кнопочности».
2. **CTA ведут наружу.** Главные действия — это переходы в **Сегодня** (выполнение),
   **Чат** (изменения плана), **Профиль** (согласие/документы), а не операции на экране.
3. **«Менять через чат».** На Workouts и Nutrition — постоянный мягкий баннер вверху
   (`ChangeBanner`): «Это просмотр плана. Изменения вносит коуч».
4. **Ревизии и «почему».** Версия плана (v8…) + **причина изменения** — это смысловой,
   а не технический блок (`RevisionFacts`). Показывать понятным языком.
5. **Пустые состояния — норма.** Новый пользователь видит много «пока недостаточно
   данных». Пустое должно **мотивировать начать**, а не выглядеть сломанным.
6. **Частичная деградация (Longevity).** Если часть секций не загрузилась — экран не
   падает целиком: вверху плашка `PartialBanner`, упавшая секция показывает `SectionError`,
   остальное живёт.
7. **Согласие (consent).** Wellness-сигналы устройств и документы завязаны на согласие.
   Нужны состояния «нужно подключить» и «согласие отозвано» → ведут в Профиль.
8. **Без медицины.** Везде, где настроение/стресс/документы/сигналы — оговорка
   `MedicalNote`: «это не клиническая оценка / только метаданные». Запрещён язык
   диагнозов, дозировок, «индексов готовности как истины».
9. **Тема и каркас.** Все три экрана — **тёмная тема**; общий каркас `AppShell` с
   сайдбаром **244px**.

## Общие компоненты состояний — `app/states.jsx`
Перенесите как переиспользуемые. Все тёмные.

- **`Sk / SkLines / SkCard`** — skeleton-блоки с shimmer (keyframes `htShimmer`,
  `linear-gradient(90deg, rgba(255,255,255,.04) 25%, .10 38%, .04 60%)`, `background-size 200%`,
  анимация 1.5s linear infinite). `SkCard` имитирует карточку (опц. шапка).
- **`Spinner`** — SVG-кольцо, `htSpin` 0.9s.
- **`LoadingScreen({label, layout})`** — единый full-screen плейсхолдер на экран
  (`layout: 'longevity' | 'plan'`): строка «{label}…» со спиннером + skeleton-карточки в
  раскладке экрана.
- **`ErrorScreen({title, msg, retry, secondary})`** — центр-блок: красный иконный бэйдж
  (`info`, `redDim`), заголовок, текст, кнопки «Обновить» + (опц.) «Открыть чат».
- **`PartialBanner`** (Longevity) — amber-плашка «Некоторые секции не обновились…» +
  «Обновить →».
- **`SectionError({label, h})`** — пунктирная заглушка одной упавшей секции внутри живого
  экрана + «Обновить →».
- **`ChangeBanner`** (Workouts/Nutrition) — indigo-плашка «Это просмотр плана. Изменения
  вносит коуч — расскажите в чате» + кнопка «Открыть чат».
- **`DailyExecCard({icon, color, title, text, cta})`** — цветная карточка-CTA «выполнение
  на Сегодня» (кнопка ведёт в Today).
- **`RevisionFacts({rev, when, source, why, accent})`** — карточка «Почему именно эта
  версия»: accent-полоса сверху, чип «v8 · активная», текст-причина, ряд фактов
  Обновлено/Источник/Версия.
- **`CoachNotes({children})`** — аватар коуча + «Заметка коуча» + текст.
- **`MedicalNote({children})`** — строка с info-иконкой, `mut2`, «не клиническая оценка».
- **`RevisionHistory({open, rows})`** — сворачиваемый список версий; активная подсвечена
  (blue) + чип «активная»; футер-пояснение «прошлые тренировки привязаны к версии, что была
  активна тогда». `rows`: `{rev, when, note, active}`.
- **`Expander({icon, title, hint, open, children})`** — общий аккордеон (для «продвинутых
  инструментов»).
- **`ConsentSignals({state})`** — wellness-сигналы устройств, **3 состояния**:
  - `data` — список (Сон/Пульс покоя/Шаги) + `MedicalNote` + «В Профиль →».
  - `connect` — пунктирная заглушка «Подключите источники данных» + кнопка в Профиль.
  - `revoked` — «Согласие отозвано» (amber, lock) + «Управлять согласием».
- **`MediaCard({kind, color, icon, title, meta, duration, tags, poster, done})`** —
  **ключевая «смотрибельная» карточка** (упражнение или рецепт). Постер 132px:
  детерминированный градиент по `poster`-индексу + крупная полупрозрачная иконка + по центру
  **`PlayBadge`** (полупрозрачный круг с треугольником) + бэйдж длительности (top-right) +
  опц. чип «готово» (top-left). Ниже: title 14/700, meta, чипы-теги.
- **`PlayBadge({size, color})`** — круглая play-кнопка для постеров/плееров.

## Interactions & Behavior
- **Все CTA ведут наружу**: «Сегодня» (выполнение/логирование), «Чат» (изменения плана/
  обсуждение), «Профиль» (согласие/документы/устройства). На самих экранах операций нет.
- **Открытие видео**: клик по `MediaCard` (упражнение) → `WorkoutsScreen state="video"`;
  клик по `MediaCard` (рецепт) → `NutritionScreen state="recipe"`. «Назад к плану» — обратно.
- **Сворачиваемые блоки**: `RevisionHistory` и `Expander` (продвинутые инструменты),
  раскрывашка отложенных доменов в `CrossDomainTrends`. Хранить `open`-состояние локально.
- **Асинхронные состояния обязательны для каждого экрана**: ЗАГРУЗКА → ПУСТО/SPARSE →
  ОШИБКА → ГОТОВО. Для Longevity дополнительно **ЧАСТИЧНАЯ ОШИБКА**; для панели следования
  Nutrition — собственные загрузка/ошибка/пусто **внутри панели**, независимо от экрана.
- **Согласие**: `ConsentSignals` и документы — переключение data/connect/revoked ведёт в
  Профиль; без согласия данные не показываем.
- **Переходы**: спокойные fade/slide ~150–200ms. `prefers-reduced-motion` → конечное
  состояние без анимации (важно для skeleton/shimmer — допускать отключение).

## State Management
- `screen.status`: 'loading' | 'sparse'|'empty' | 'error' | 'partial'(longevity) | 'ready'.
- `longevity`: consistency%, дни недели[], домен-сводки, цели[], чек-ины 7д (mood/stress),
  consent{signals}, документы[] (метаданные), кросс-домен паттерны[] + отложенные домены[],
  флаги «какие секции упали» (для partial).
- `workouts`: activePlan{name, desc, rev}, неделя[], сессия дня + упражнения[]
  (видео-метаданные), revisionFacts{rev, when, source, why}, coachNote, history[],
  weeklyProgress{stats, adaptationPack, tools}.
- `nutrition`: activePlan{name, desc, rev}, goals{kcal, protein, carbs, fat}, meals[],
  prefs/restrictions/allergies, coachNote, **adherence{status, meals[], protein, water}**
  (своя загрузка/ошибка), recipes[] (видео-метаданные), history[].
- `media`: выбранное упражнение/рецепт для плеера; плейсхолдеры постеров → реальные
  видео/изображения в проде.

### Как открыть референсы локально
React/Babel грузятся с CDN → нужен статический сервер (CORS на `file://`):
```
cd design && python3 -m http.server 8000
# http://localhost:8000/index.html  — весь канвас
# http://localhost:8000/preview.html?s=longevity  — один экран
```
> `*.jsx` компилируются in-browser через Babel (это прототип). В проде — нормальная сборка:
> атомы/состояния/экраны переносите как компоненты, токены — как CSS-переменные / тему.
