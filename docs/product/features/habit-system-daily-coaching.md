# Habit System and Daily Coaching

**Status:** Proposed - Today checklist and `create_today_checklist` proposals exist; durable habit state, recurrence, completions, and adherence do not.

## Objective

Create a durable Habit domain that lets the coach propose wellness habits, stores the approved habit plan as revision-safe structured state, materializes scheduled habits into Today, and reports recent adherence for coaching and Longevity.

Habits are not a primary navigation surface. Chat proposes and explains changes, Today is the daily execution surface, Longevity shows consistency trends, and Profile may expose relevant preferences or consent.

## Non-Goals

- No diagnosis, treatment, clinical sleep protocol, clinical mental-health protocol, medication guidance, or medical certainty language.
- No automatic habit creation or plan mutation without user approval.
- No manual habit-plan editor in Today or Longevity for MVP; durable plan changes happen through Chat proposals.
- No device-sync auto-completion, reminders, push notifications, badges, leaderboards, or social accountability in MVP.
- No full scheduling engine, habit stacking engine, or cron-like recurrence language.
- No replacement for nutrition targets, workout plans, or `create_today_checklist`; habits may link to those domains when useful.

## Product Rules

- Structured state is authoritative. Chat history and Today checklist labels are not the source of truth for habits.
- Habit plan changes create immutable revisions; they never overwrite active plan payloads in place.
- Today completion actions are execution events, not plan edits.
- AI may propose habit creation/adaptation, but backend validation and user approval apply the change.
- Wellness copy may recommend general behavior prompts such as walking, hydration, sleep routine, mobility, journaling, or breathing practice, but must not frame them as treatment.
- Consented documents or device metrics can influence proposals only after the relevant consent feature exists and explicit consent is recorded.

## In Scope

- One active `HabitPlan` per user with immutable `HabitPlanRevision` records.
- Habit definitions stored inside the revision payload with stable definition IDs and typed metadata.
- MVP recurrence for `daily` and `selected_weekdays`; richer cadence rules are deferred.
- MVP targets for `boolean`, `count`, `duration_minutes`, and `numeric`.
- Today materialization for scheduled habits with stable source refs.
- Completion logging for `complete`, `skipped`, and `missed` states.
- Basic adherence reads: 7-day rate, 30-day rate, current streak, scheduled/completed counts.
- Typed AI proposals for `create_habit_plan` and `adapt_habit_plan`.
- Optional starter template catalog for safe, common habit suggestions.

## User Stories

- As a user, I can ask the coach to suggest daily habits and approve the plan before it becomes active.
- As a user, I can open Today and see the habits scheduled for the selected date.
- As a user, I can complete or skip a habit from Today and trust that it contributes to habit history.
- As a user, I can ask the coach to pause, remove, or adjust habits through an approved proposal.
- As a user, I can see recent consistency without losing history when a habit is revised or paused.

## Data Model Decisions

### Tables

| Entity | Purpose |
| --- | --- |
| `habit_plans` | Stable per-user plan identity: `id`, `userId`, `status`, `activeRevisionId`, timestamps. |
| `habit_plan_revisions` | Immutable revisions: `id`, `habitPlanId`, `revisionNumber`, `reason`, `source`, `payload`, `createdAt`, optional `proposalId`. |
| `habit_completions` | Execution facts: `userId`, `habitDefinitionId`, `localDate`, `status`, `progressValue`, `unit`, `todayChecklistId`, timestamps. |
| `habit_templates` | Optional seeded catalog for starter habits and AI validation. |

### Stable IDs and Linkage

- `habit_plans.id` is stable for a user until the plan is archived.
- Each habit definition has a stable `habitDefinitionId` generated by the backend when a proposal is applied.
- `adapt_habit_plan` must preserve `habitDefinitionId` for the same conceptual habit across revisions, including title, target, schedule, optionality, or display-order changes.
- A genuinely new or replacement habit receives a new `habitDefinitionId`; removed or paused habits keep historical completions.
- Today habit items store `source: { type: "habit", habitPlanId, habitPlanRevisionId, habitDefinitionId }`.
- Materialization is idempotent by `(userId, localDate, habitDefinitionId)`. Re-running sync for a date must not duplicate Today items.
- `habit_completions` is unique by `(userId, habitDefinitionId, localDate)`.
- Dates use the user's configured timezone to compute `localDate`.

### Habit Definition Payload

Minimum revision payload per habit:

- `habitDefinitionId`
- `title`
- `category`: `hydration`, `movement`, `nutrition_support`, `sleep_routine`, `mobility`, `mindfulness`, `planning`, `other`
- `status`: `active`, `paused`, `removed`
- `schedule`: `{ type: "daily" }` or `{ type: "selected_weekdays", daysOfWeek: number[] }`
- `target`: `{ type, value?, unit? }`
- `required`: boolean
- `timeOfDayHint`: optional `morning`, `midday`, `evening`, `anytime`
- `linkedSource`: optional domain ref, initially `nutrition_hydration_target` or `workout_movement_context`
- `coachingNote`: optional wellness-safe text
- `displayOrder`

### Resolved Semantics

- Active habit cap: maximum 12 active habits per plan in MVP.
- Required habits count toward daily adherence and streaks.
- Optional habits appear in Today but do not reduce adherence or break streaks when skipped or missed.
- For required habits, `complete` continues streak, `skipped` and `missed` break streak.
- Hydration habits should use `linkedSource: "nutrition_hydration_target"` when an active nutrition hydration target exists; the habit reinforces execution without redefining the nutrition target.
- Walking or movement habits may reference workout context but do not create or edit workout sessions.

## API and Tooling Boundaries

### Backend APIs

- `GET /habits/plan` returns the active plan and active revision payload.
- `GET /habits/plan/revisions` returns revision metadata and payloads for audit/history.
- `GET /habits/adherence?window=7|30` returns per-habit and plan-level adherence.
- `POST /today/:date/sync` or the existing Today read path materializes habit items for the selected date.
- Existing Today completion endpoint updates the checklist item and delegates habit-source completions to `HabitsService`.

### Proposal and AI Tools

- Add proposal intents `create_habit_plan` and `adapt_habit_plan`.
- `create_habit_plan` creates the first active plan revision after approval.
- `adapt_habit_plan` creates a new revision for add, pause, remove, target, schedule, order, or note changes after approval.
- `create_today_checklist` remains date-specific. It must not create durable habits and should avoid duplicate labels for habits already scheduled that day.
- AI context may include active habit summary and recent adherence only from structured reads.
- Backend validation rejects unsafe language, unsupported target types, unsupported recurrence, excessive targets, more than 12 active habits, unknown template IDs, and conflicts with hard user constraints.

### Frontend Boundaries

- Chat renders habit proposal cards with approve/reject actions and post-approval state refresh.
- Today renders habit-sourced checklist items using existing checklist patterns plus source-aware completion.
- Longevity can show adherence summaries after the backend read model exists.
- Profile remains limited to preferences/context; no habit-plan editor in MVP.

## Acceptance Criteria by Slice

### Slice 1: Domain Foundation

- A user can have at most one active habit plan.
- Applying an accepted habit proposal creates a `habit_plans` row if needed and a new immutable `habit_plan_revisions` row.
- Active revision reads return only validated payloads owned by the authenticated user.
- Rejected and pending proposals do not create or update habit plans.
- Existing workout, nutrition, and Today checklist flows continue to work.

### Slice 2: Today Materialization and Completion

- Today includes habit-sourced checklist items for habits scheduled on the selected local date.
- Sync is idempotent and does not duplicate items for the same date and habit definition.
- Habit-sourced Today items retain source refs to plan, revision, and definition IDs.
- Completing or skipping a habit item updates the checklist item and upserts one `habit_completions` record.
- Paused or removed habits do not materialize for future dates but historical Today items and completions remain readable.

### Slice 3: Proposal Pipeline and Chat UI

- Chat can produce, persist, render, accept, and reject `create_habit_plan` and `adapt_habit_plan` proposals.
- Accepted proposals preserve stable definition IDs for unchanged habits and assign backend IDs for new habits.
- Validation errors are surfaced without mutating habit state.
- Proposal copy and habit notes stay in wellness/coaching language.

### Slice 4: Adherence and Longevity Read Model

- API returns 7-day and 30-day adherence per habit and plan-level summary.
- Current streak follows the required/optional semantics in this brief.
- Longevity or a lightweight summary surface can render consistency without exposing a new primary tab.
- AI context includes active habit summary and recent adherence from structured state.

### Slice 5: Templates and Guardrails

- Starter habit templates are seeded or declared in code with safe categories, target ranges, and copy.
- AI proposal validation can accept known template references and reject unknown or unsafe ones.
- Hydration and movement templates respect nutrition/workout linkage rules.

## Test Strategy and Verification Matrix

| Area | Tests | Runtime verification |
| --- | --- | --- |
| Schema and contracts | Zod accepts valid payloads and rejects unsupported schedules, targets, unsafe copy, too many active habits, and unknown template IDs. | Invalid proposal shows validation error and leaves state unchanged. |
| Revision safety | Accepted create/adapt creates new revisions; old revisions remain unchanged; pending/rejected proposals do not mutate state. | Approve an adaptation and confirm history plus active revision update. |
| Stable IDs | Unchanged habits keep `habitDefinitionId`; new habits get backend IDs; removed habits keep historical completions. | Adapt title/target, then confirm Today and adherence still link to prior completions. |
| Today sync | Scheduled habits materialize once per local date; paused/removed habits stop future materialization. | Load Today twice for same date and verify no duplicates. |
| Completion | Complete/skip upserts one completion and updates checklist state. | Mark a habit complete, refresh, and confirm completion persists. |
| Adherence | 7/30-day rate and streak calculations match required/optional semantics. | Complete several days in fixtures and confirm returned summary. |
| Cross-domain links | Hydration habit links to nutrition hydration target when present and does not redefine macros/hydration. | Active nutrition hydration target appears as habit execution prompt, not a new target. |
| Safety and auth | User can read/write only own habits; copy avoids medical claims. | Attempt cross-user access in API tests; review visible copy. |
| Regression | Existing workout Today sync and `create_today_checklist` behavior remain green. | Existing Today workout/checklist smoke flow still passes. |
